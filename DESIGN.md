# Okami Maturity Design Direction
## Visual Thesis

Okami Maturity should feel like a serious AppSec/product surface: dark, precise, technical, and premium. The page should sell measurable roadmap and evidence, not spectacle. Use Okami DS for SAMM 5 functions / 15 practices / 90 questions.

## Palette

- Void black: `#060609`, primary canvas
- Panel black: `#0b0b12`, product surfaces
- Signal orange: `oklch(72% 0.19 45)`, primary action and execution
- Control magenta: `oklch(70% 0.27 340)`, risk and intervention
- Runtime cyan: `oklch(82% 0.14 200)`, live state and docs
- Success green: `oklch(75% 0.17 145)`, connected state

## Typography

- Display: Space Grotesk, restrained and heavy only for the main thesis
- Utility/data: JetBrains Mono for labels, routes, commands and state
- Avoid oversized type inside utility sections. Cards and panels should use compact headings.

## Layout Rules

- Hero copy must answer the buying question in 5 seconds.
- Product surfaces should look like systems, not generic cards.
- Use numbers only when order or count matters.
- Runtime, company and related products must not be grouped as one ecosystem.
- Footer stays simple: logo, short positioning, essential links, copyright.

## Signature Element

The page uses a "mission evidence rail": compact proof strips and terminal-like state surfaces that show what Okami Maturity delivers: execution, cost, risk, queue and runtime boundary.

## Copy Rules

- Speak from the buyer side: evidence, control, cost, risk, runtime boundary.
- Avoid broad claims like "revolutionary" unless the page proves the mechanism.
- Keep CTAs concrete: "View cockpit", "Read architecture", "Audit source".
- Do not expose client infrastructure details or name private runtime internals.
